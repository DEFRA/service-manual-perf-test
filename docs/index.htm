1          Introduction

The purpose of this Performance Testing Strategy document is to describe at a high level the Digital Data Technology Services (DDTS) strategic approach to performance testing across all programmes.

Key objective for performance testing is as below:

·         Validate that the developed solution complies with the approved non-functional requirements (NFRs) and contributes towards the Government Digital Service (GDS) service standards mandate.

·         Ensure any performance issues (If any) identified during execution are documented through the defect process and remediated (aligned to entry/exit criteria) in solution before going live.

·         Validate the performance of the frontend and back-end API’s /microservices.

Performance testing is a non-functional software testing technique that determines how the stability, speed, scalability, and responsiveness of an application hold up under a given workload. This includes testing the application to verify the below points.​

  • How does the application including each microservice and end-to-end flow perform under normal Load? ​
  • How does the application behave when the expected number of users log in concurrently? ​
  • Can the application handle the equivalent load during peak hours/day?​
  • How is the application performance affected when exercised for long durations (e.g., 24 hours continuous use)?
  • How is the application perfor​mance affected as the number of concurrent users exceeds the average expected number?

·         To identify system breaking point and failures from stress testing.

Exhaustive testing of all functions, scenarios and variants is impossible and attempting it is both expensive and time consuming and does not add value

For this reason, Performance Testing effort should be Risk Based aligned with DDTS Test Strategy and will focus on following areas:

·         Critical business processes

·         High volume business transactions which simulate real world scenarios

·         Visible to customers and specifically externally facing customers

·         Expose Defra to risk

·         Critical to functions for delivery

·         Complex from a technology point of view

·         Complex from a business point of view

The detail of exactly what will be tested will be defined based on a Volumetric Analysis process and documented in the Test approach. The Performance test resource will work with Business Analysts, Solution Architects, Devs and QA Leads, CCOE and others to establish and agree this content aligned to NFR’s.

2          Test Approach

This section will detail the performance testing approach, the type of tests to be conducted during performance test and the pre-requisite for each test.

Figure 2.2: Depicts an example of EU Trade Performance test timeline view

 

 

Early Performance (Shift-Left) Testing: It is expected that suppliers and internal team who are developing the solution do their initial performance checks on the code in lower environment to ensure it is performant and DDTS performance test resource can support this.

 

Integrated: This will be done by the allocated Performance test resources in DDTS QA & Test.

 

Note: Early and Integrated performance testing will validate both microservice SLAs and end-to-end journey performance requirements.

 

The following are the performance test types recommended for any newly built services.

 

1.       Normal load test

2.       Peak load test

3.       Endurance/Soak test

4.       Stress Test

2.1            Normal Load Test:

Tests shall be conducted to evaluate the performance of the System under Test using normal day user load and transactional volume.​ This will be based on analysis conducted to determine what is the average user load, where known and supported by the NFR’s.

2.2           Peak Load Test

Tests shall be conducted to evaluate the performance of the System under Test using peak loads of users and/or transactions.​​ This will be based on analysis conducted to determine what the peak user load was previously. For new applications this will evolve and be subject to change until a stable product is baselined by the Business.

2.3         Endurance/Soak Test

Tests shall be conducted over an extended period (usually at least 10 hours) to check that the System under Test (SUT) does not degrade in its performance and to assess the system’s stability after it proceeds into operation.

 

As the objective of the test, it is to verify the stability under a sustained load without pushing it through unexpected load thresholds, a 60% of the expected Peak load per hour is applied sustained along the duration of the test.

2.4         Stress Test

Tests shall be conducted to operate the System under Test (SUT) beyond its design limits to identify any potential breaking points or bottlenecks and assess the failures. During stress tests usually the peak volumes are continually doubled. We will escalate until breakage or concurrency is deemed excessive.​

2.5         Resilience test support

Supporting infrastructure teams during Resilience Testing activity such as but not exhaustive, component outage, availability zone down, chaos testing, and recovery processes.

 

2.6 Production feedback Loop and Observability

Continuous Production feedback on NFR and Test coverage would keep performance test up to date and simulate real world scenario which would provide realistic test results and prevent defect leakage to production.  

Monitoring dashboards should be setup in Production using Logs and Metrics from various sources like Azure Monitor, Google Analytics, Open Search, Grafana etc. This will help in improving observability, Defect detection and production feedback.

4          Monitoring

It is essential that projects setup monitoring for example dashboard in azure portal and add performance test environment application and database servers’ resources to the dashboard. This enables performance test teams to have all resources in one place to monitor and understand how the application and database servers perform during test execution. And it also allows to grab the test evidence to be captured & stored after each test

6.3         Defect Management Tool

Preferred Defra tool for agile development is Azure Dev Ops from Quality Assurance Perspective, its recommended to use ADO for performance test user stories and use the same for defect management, documenting the detail plan and test results. However, its accepted that projects may be using/used Jira.

6.4         Test Result Reporting

The results will for example include details such as:​

  • Test passed/Failed? And where if it has failed?​
  • Response times for each request, is this within the SLA defined in the Performance NFRs?​
  • Actual transactions passed against expected numbers.​
  • Number of transactions failed and the error messages.​
  • What % of the passing transactions are meeting the SLA?​
  • Server stats will be formally captured if failure scenarios are identified.

7          Environment

It is recommended to carry out performance testing in a production sized dedicated test environment for test run so that results reflect what you would expect to see in production.

However, in some instances due to the nature of the project this may not be possible. In this case, the project needs to list the difference between production environment and the environment that is provisioned for performance testing, and the associated risk must be agreed by the business and project.

Where the performance test environment is not like production, we also need to adjust the test volumes accordingly based on environment sizing.

8          Status Reporting

Regular progress report will be sent out to the programme by QA leads and will be shared with technical and business stakeholders. And the frequency of reporting will be agreed with stakeholders.

This will include:​

  • RAG Status​
  • Summary of weekly progress and future planned activities​
  • Key milestones​
  • Issues impacting testing ​
  • Number of tests planned. ​
  • Number of tests completed.​
  • Number of tests that have failed. ​
  • Number of tests that are blocked.​
  • Progress against the plan
